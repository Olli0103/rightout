# Independent release review: v0.3.0

Review date: 2026-07-12. Scope: the complete `feature/live-removal-v0.3.0` working tree, official OpenClaw `2026.6.11` package documentation/types/runtime, official Brave and broker/provider sources, packaging, tests, and live-write boundaries. All network/write behavior in tests used mocks and `.invalid` identities; no real scan, email, or broker request was performed.

## Findings and resolutions

| ID | Severity | Finding | Resolution |
| --- | --- | --- | --- |
| RO-030 | P0 | Sanitized SMTP failures retained the raw provider exception as `cause`, exposing PII/credentials through logs or inspection. | Removed the cause chain; adversarial tests inspect message, stack, own properties, and recursive serialization. |
| RO-031 | P1 | Removal pre-approval validation initially resolved and parsed private profile/SMTP values. | The hook is PII-free; SecretRefs are resolved/validated only after `allow-once` and before network access. |
| RO-032 | P1 | Removal approval was not bound to the exact profile and SMTP snapshots. | Added operator-generated normalized SHA-256 bindings, action-bound approval scope, and post-approval/pre-network rechecks. |
| RO-033 | P1 | Scan approval allowed a different private subject payload under the same opaque profile ID. | Added recorded scan consent, operator consent review, normalized scan-profile digest, and pre-network substitution rejection. |
| RO-034 | P1 | `US-CA` eligibility trusted a free-form jurisdiction array even when country/region contradicted it. | The live lane now requires `country=US`, `region=CA`, and consistent `US` plus `US-CA` tags; contradictory fixtures perform zero sends. |
| RO-035 | P1 | Microsoft 365 was advertised with Basic SMTP credentials after its OAuth migration boundary. | Removed Microsoft 365 from code, manifest, tests, and docs; it remains unsupported until a separately reviewed OAuth 2.0 flow exists. |
| RO-036 | P1 | Scan consent existed as a documentation claim but was not a technical gate. | The private profile must contain non-future recorded consent with `scan` scope, and operator attestations must record consent review. |
| RO-037 | P1 | The installer test fixture lagged behind the strengthened scan attestation schema. | Updated the isolated OpenClaw config fixture; fresh and forced runtime installs pass. |
| RO-038 | P1 | Nodemailer used a semver range while the published archive omitted the lockfile, so installed code could differ from tested/SBOM code. | Exact-pinned `9.0.3`; release validation now cross-checks every production dependency against package-lock and both SPDX SBOMs. |
| RO-039 | P1 | OpenClaw rolling documentation and the pinned stable package differed around `timeoutBehavior`. | The release follows the pinned stable `2026.6.11` docs/types/runtime and explicitly sets `deny`; conformance docs state the version boundary. |
| RO-040 | P1 | A consent-aware worst-case scan approval description exceeded the local 256-character release bound. | Shortened without removing action, target, disclosure, retention, consent, or risk facts; boundary test passes. |
| RO-041 | P1 | An already-aborted signal could race between the last pre-send check and SMTP listener registration. | SMTP checks before transport creation and again after listener registration; aborted calls create no transport. |
| RO-042 | P1 | The Brave POST live contract needed a direct authoritative source because general quickstart examples emphasize GET. | Pinned the official Web Search POST API reference and exact endpoint/body behavior in docs, tests, and release validation. |

## Review outcome

The independent closing review reported **no open P0/P1** in the current candidate. P2/P3 product gaps are documented rather than hidden: process-local rather than durable dedupe, no universal consent max-age/revocation registry, one automated removal lane, no mailbox verification, no durable live case ledger/dashboard, no scheduled rechecks, no screenshots/direct absence proof, no custom URL removals, and no broad/private-broker coverage. Profile/SMTP digests are pseudonymous sensitive metadata, not anonymous data.

Final GO remains conditional on the full matrix, protected-branch PR/CI, merged-commit tag CI, published checksums/assets, and an explicit final independent review result.
